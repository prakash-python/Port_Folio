The Pipeline: When a user uploads a financial document (e.g., a PDF bank statement), the Django API immediately returns a `job_id` so the UI isn't blocked. Redis queues the task. A Celery worker picks it up, runs it through PaddleOCR for text extraction, and then pushes the markdown to an LLM (OpenAI/Groq). The LLM parses the unstructured text into a structured JSON array of exact transactions. Finally, the worker fires a webhook callback to update the main database, and the React UI polls/receives the completed status to render the analysis dashboard.

1. AI-Powered Document Intelligence Microservice (AutoDoc-Validator)

Python Django DRF Celery Redis PaddleOCR LLM (OpenAI)
View Code
The Problem

Manual verification of loan documents (bank statements, payslips) is an extremely slow, error-prone bottleneck delaying loan approvals. Rule-based parsers fail because every bank (SBI, HDFC, ICICI) formats statements differently.

The Solution

I built a standalone microservice that asynchronously ingests documents, runs OCR, and utilizes Large Language Models to intelligently normalize transactions across *any* bank format into structured JSON, firing a webhook callback when complete.

My Contributions & Features
  • Async Pipeline: Engineered the Celery worker queue to prevent 1-2 minute OCR jobs from blocking API requests.
  • LLM Normalization: Integrated OpenAI to bypass complex regex parsing, achieving high accuracy in extracting transaction tables from unstructured OCR markdown.
  • Webhook Callback: Developed robust background delivery mechanism with exponential backoff retries to guarantee result delivery to the main LOS.
  • Concurrency Locks: Implemented PostgreSQL select_for_update locks to prevent duplicate Celery workers from racing on the same job.

2. Insta EMI Core Backend (Loan Origination System)

Django PostgreSQL Celery JWT AES-256 Encryption Groq API
View Code
The Problem

Lending involves 9 different employee roles (sales, verification, credit). A monolithic database schema without strict boundaries leads to data leaks. Financial data (PAN, Income) must be encrypted at rest per compliance.

The Solution

A highly modular Django REST API comprising 14 apps, segregating namespaces by role. Leveraged JWT claims for stateless role verification, and PyCryptodome for transparent field-level AES encryption.

My Contributions & Features
  • 9-Tier Role Architecture: Enforced rigorous data isolation so a Process Associate and a Regional Sales Manager access entirely different views of the same application.
  • Data Security: Implemented transparent AES-256 encryption on Django model fields safeguarding PAN, Aadhaar, and income figures.
  • Integrations: Built API boundaries to connect with live CIBIL bureau APIs, AWS/Azure Blob storage, and the standalone AuthDoc Validator service.
  • Fuzzy Matching: Used FuzzyWuzzy for intelligent matching of entered company names against the employer master list.
